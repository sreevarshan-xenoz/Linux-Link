6. Wiring & Bug Audit

🔴 Critical Bugs (All Fixed)

BUG-1: Android control reader silently discards all server packets
File: android/rust/src/api.rs
Fixed in commit `9486f76`: Reader task now parses incoming lines as KDE Connect packets and forwards them through a broadcast::Sender. Flutter polls them via poll_incoming_packets(). Notifications now reach the Android notification service.
BUG-2: Notification monitor captures notifications but they never reach any client
File: server/src/service.rs, server/src/notification_monitor.rs
Fixed in commit `a0bfaee`: Added ACTIVE_CLIENTS static Vec to track connected KDE Connect writers. Spawned a notification broadcast task that subscribes to the notification channel and pushes kdeconnect.notification packets to all active clients. Dead writers are pruned on send failure.
BUG-3: Streaming pipeline only uses PipeWire, never falls back to X11
File: core/src/streaming/streamer.rs:150
Fixed in commit `18bb825`: Changed from capture::start_capture() to capture::start_capture_auto() which auto-detects the display server and falls back to X11 via xcap when PipeWire is unavailable.
BUG-4: CertManager is in-memory only on Android — trusted certs lost on restart
File: android/rust/src/api.rs
Fixed in commit `9486f76`: Added set_data_dir() FFI to configure a persistent path from Flutter. connect_streaming() now calls CertManager::load_or_create() with that path. Added list_trusted_peers() and forget_trusted_peer() FFI functions for the trust screen UI. Added known_peers() and remove_peer() accessors to the core CertManager struct (transport.rs).

🟡 Warnings & Design Issues

WARN-1: Streaming stats always return zeros
File: android/rust/src/api.rs:472-480
`get_streaming_stats()` hardcodes `fps: 0.0`, `bitrate_kbps: 0`, `frame_drops: 0`. The streaming client has no metrics tracking. The health provider and stats overlay in Flutter will always show zeros.
WARN-2: Audio streaming is silence (PipeWire loopback not implemented)
File: core/src/streaming/streamer.rs:452-533
Audio task 6 encodes silence PCM frames. The TODO comment says "Replace with PipeWire audio loopback capture." Audio streaming (F1) is not functional.
WARN-3: Parallel input paths may conflict
Files: android/rust/src/api.rs send_mouse_event/send_keyboard_event (QUIC) vs server/src/plugins/input.rs (KDE Connect TCP)
Android sends input via QUIC when streaming is active, but falls back to KDE Connect TCP. The server has TWO input receivers: (1) Streamer Task 4 reads QUIC streams and injects via `handle_input_packet()`, (2) KDE Connect InputPlugin reads TCP packets and injects via `InputInjector`. These use different key mapping functions (`android_to_evdev_keycode` vs `key_name_to_enigo_key`), leading to inconsistent behavior.
WARN-4: Clipboard get/send opens new TCP connections
File: android/rust/src/api.rs:175,202
`send_clipboard()` reuses the existing control channel (`CONTROL_WRITER`), but `get_clipboard()` opens a new independent TCP connection. This is inconsistent and doesn't respect the existing connection state.
WARN-5: `StreamingClient::new()` has dead code — channels dropped immediately
File: core/src/streaming/client.rs:56-68
The constructor creates `mpsc::channel` but drops the receivers. New channels are created in `connect()`. The `new()` function is unused in production.
WARN-6: `is_streaming_active()` uses `try_lock()` — race condition
File: android/rust/src/api.rs:457-462
Uses `try_lock()` which returns false if the mutex is briefly held by another task. Can report false negatives.
WARN-7: Device identity is ephemeral (PID-based)
File: server/src/kde.rs:49
`host_device_id()` uses `format!("linux-link-{}", std::process::id())`. Every server restart generates a new device ID. KDE Connect pairing is based on device IDs, so all pairings break on restart.
WARN-8: `receive_frames`/`receive_audio` timeout applies only to first packet
File: android/rust/src/api.rs:531-593
The `timeout_ms` parameter only gates the first `recv()`. After that, `try_recv()` is used with no timeout. This is correct for draining but the timeout semantics may confuse callers.
WARN-9: Raw TCP helpers in Dart bypass Rust FFI trust/cert verification
File: android/lib/rust_api_bridge.dart:234-387
`sendPowerCommand()`, `executeCommand()`, and `getMonitorCount()` open raw TCP sockets from Dart and implement the LINUX_LINK_HELLO handshake manually. These bypass the Rust `ConnectionManager` which does proper handshaking. They also use short, hardcoded timeouts that may fail on slower connections.
WARN-10: mDNS `register_service` may fail silently
File: core/src/tailscale/lan.rs:51-77
If mDNS registration fails, the error is logged but the server continues. The Android client's `scan_lan_once()` will just not find this server. No alert is surfaced to the user.

🔵 Info & Notes

INFO-1: Protocol version is hardcoded, not linked to crate version
`HANDSHAKE_HELLO` uses "1" while crate version is "0.1.0". These are independent — fine for now.
INFO-2: `get_monitor_count()` opens separate TCP connection
File: android/rust/src/api.rs:486-523
Creates a fresh `ConnectionManager` + TCP connection just to query monitors. This is fine for a one-shot query but adds latency.
INFO-3: FFI `send_mouse_event` special-cases button=2 as scroll
File: android/rust/src/api.rs:614-619
Button 2 is reserved for scroll events on the Flutter side. The binary protocol has separate `MouseScroll` and `MouseClick` variants.
INFO-4: Gamepad input → keyboard/mouse mapping in injector
File: server/src/input_injector.rs:283-329
Gamepad axes/buttons are mapped to keyboard/mouse for broad compatibility. Left stick→mouse, A→Enter, B→Escape, DPad→arrows.
INFO-5: Enigo → uinput fallback chain for input injection
File: server/src/input_injector.rs:44-55
enigo (XTEST) works on X11/XWayland. uinput works on all compositors but requires `/dev/uinput` access.
INFO-6: TailscaleClient has 4-tier fallback chain
File: core/src/tailscale/mod.rs
CLI → LocalAPI (3 ports) → Android HTTP API → netcheck → empty response.
INFO-7: 2 capture paths + 2 input paths + 2 transport paths = combinatorial complexity
Capture: PipeWire (Wayland) | X11 (xcap/x11rb)
Input: enigo (X11) | uinput (kernel)
Transport: TCP (KDE Connect) | QUIC (Streaming)
Each path may have different behavior, error handling, and capabilities.
